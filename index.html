考選部 PDF：試題 + 答案兩份

  • 每梯次 4 份卷（醫學三、四、五、六），每卷各 80 題
  • 試題 PDF（Q，約 1MB）：題文 + 選項 A-E
  • 答案 PDF（A，約 50KB）：含「備註」欄列出更正題與送分說明
  • 每年考後可從考選部網站下載

parse-exam.py：從 PDF 提取結構化資料

  • 以 pdfplumber 讀取原始文字，過濾頁首頁尾雜訊
  • 連號題號辨識（「3.」之後必須是「4.」），防止正文內的年份「1994.」誤觸發
  • 選項 A→B→C→D→E 連續辨識，換行文字自動黏接
  • 答案備註解析：「答Ａ或Ｃ者均給分」→ "AC"；「一律給分」→ "ALL"
  • 啟發式題型標記：含「如圖/下圖」→ image；含「病人/患者/歲」→ clinical；其餘 → knowledge
  • 解析失敗直接中止報錯（不產出半套資料），降低人工除錯成本

data/exams/*.json：版本控管的權威來源

  • 命名規則：115-1-3.json（115 年第 1 次醫學三）
  • 這份 JSON commit 進 git — 之後無論資料庫重建幾次，只要重跑 import-exam.ts 就能還原
  • 若 PDF 解析後有 OCR 錯字，直接改這個 JSON 再重跑匯入，不用動資料庫
  • 同時有 115-1.tags.json：AI 標記的科別對照表（可 commit 供人工審閱）

import-exam.ts：冪等 upsert 入庫

  • 以 (examYear, examSession, examPaper, questionNumber) 四欄位找出已存在的題目
  • 存在 → 只更新題文/選項/答案，不覆寫 specialty/topic/noteSlug（保留 AI 標記與人工修正）
  • 不存在 → 新建，specialty 預設 "未分類" 等待 AI 標記
  • 重跑完全安全，不會產生重複題目

tag-specialties.ts：把科別標記寫入資料庫

  • 讀取 *.tags.json（格式：卷代碼 → 題號 → 科別字串）
  • 按 (examYear, examSession, examPaper, questionNumber) 找到題目，更新 specialty
  • 同時把 aiConfidence 設為 0.9（表示 AI 標記，Admin 後台可修改低信心題目）
  • 多科別用逗號分隔（如 "腸胃內科,感染科"），第一個是主科別

PostgreSQL Question 表：所有題目的儲存地

  • 每一道題是一筆 row，以 cuid 字串為主鍵（id）
  • 複合索引 (examYear, examSession, examPaper)：整卷查詢極快
  • 索引 (specialty, noteSlug)：科別篩選 + 筆記連動查詢
  • correctAnswer 永遠在 DB 端，前端直接查詢的路徑不包含這個欄位（見下個節點）

Server Component：兩種查詢模式

  • 瀏覽模式（/practice/exam/[examId]）：包含 correctAnswer — 直接看答案
  • 作答模式（/practice/exam/[examId]/test）：刻意排除 correctAnswer，資料傳到前端時根本拿不到答案
  • 判分發生在 POST /api/practice/submit：Server 端從 DB 取正解，與使用者作答比對後寫入 UserAnswer
  • 這確保答案不可能被有技術能力的用戶從 Network tab 直接偷看

前端題卡：使用者看到的最終呈現

  • 題號顯示格式：115-1-3-1（年-梯次-卷代碼-題號），由 lib/quiz.ts formatQuestionCode() 組合
  • 作答信心標記：綠（有把握）/ 黃（不確定）/ 紅（用猜的）— 存在 PracticeSession.state
  • 每次作答動作即時存進 state → 跨裝置續答不丟失進度
  • 附圖題（questionType=image）目前顯示「附圖整理中」徽章；圖片走 R2 signed URL，禁止公開

Question 表：逐欄位解說

這張表是整個題庫系統的核心。以下用 115 年第 1 次醫學三第 1 題（衰弱量表題）做完整示範。

題目 1 編號：115-1-3-1 · 更正題（A 或 C 均給分）
有關衰弱（frailty）的量表或測量工具，下列何者錯誤？
ALinda Fried Physical Frailty Index，≧1分即為衰弱
BKenneth Rockwood Cumulative Deficit Frailty Index要評估許多面向，分數算法是（有缺損面向數／評估面向總數）
CVenerable Elder Survey 13工具，≧3分即為衰弱
DStrawbridge等人1994年設計的Frailty Measure，problem/difficulties in≧2 domains即為衰弱

🟠 正解：A 或 C 均可給分（更正題，correctAnswer = "AC"）

欄位名 型別 本題範例值 用途說明
idPK String (cuid) "cm1ab2cd3..." 系統主鍵。其他表（UserAnswer、QuestionBookmark）用這個 id 來「指向」這道題
examYear Int 115 民國年份。題號顯示格式 115-1-3-1 的第一段
examSession Int 1 第幾次（1 或 2）。同年可能有兩次考試。題號顯示格式 115-1-3-1 的第二段
examPaper String "醫學三" 試卷名稱，固定 4 種：醫學三/四/五/六。題號顯示時轉成數字代碼：115-1-3-1。對應的科別範圍是固定的（醫學三 = 內科 + 家醫）
questionNumber Int 1 該卷第幾題（1~80）。題號顯示格式 115-1-3-1 的第四段
questionText Text "有關衰弱（frailty）的量表或測量工具..." 題目正文。@db.Text 表示 PostgreSQL 用 TEXT 型別，無長度上限（有些臨床案例題非常長）
optionA String "Linda Fried Physical Frailty Index..." 選項 A 的文字。四個選項永遠存在
optionB / C / D String "Kenneth Rockwood..." 同上，選項 B、C、D
optionE可空 String? null 第五個選項（部分題型才有）。本題為 null，前端根據是否 null 決定顯示幾個選項
correctAnswer安全關鍵 String "AC" 正確答案。三種格式：
• 單字母 "B" — 一般題
• 多字母 "AC" — 更正題（選 A 或 C 都給分）
• "ALL" — 送分題（選任何答案都給分）
作答模式查詢刻意排除此欄位，判分一律 Server 端完成
questionType String "knowledge" 題型：knowledge（純知識）/ clinical（有病人情境）/ image（附圖題）。由解析器啟發式判斷，影響前端顯示方式
specialtyAI 標記 String "家庭醫學科" 科別。由 AI 標記，可由 Admin 手動修正。多科別用逗號："腸胃內科,感染科"，第一個是主科別（統計/雷達圖用）
topicAI 標記 String "衰弱症" 章節/主題（比科別更細）。由 AI 標記，用於精準連結到對應筆記段落
imageUrl可空 String? null 附圖的 R2 Storage key（非完整 URL）。前台一律透過 signed URL 提供（1小時有效），禁止公開 bucket。目前附圖題先存文字，imageUrl 為 null
noteSlug可空 String? "family-medicine-frailty" 連結到哪篇筆記的 slug。建立後可在筆記頁底部看到相關考題，也可從考題跳到對應筆記章節
aiConfidenceAI 標記可空 Float? 0.9 AI 標記的信心分數（0~1）。tag-specialties.ts 套用時固定設 0.9。Admin 後台可以只看低信心（<0.8）題目做人工確認
createdAt DateTime 2026-06-12T10:30:00Z 題目匯入時間。自動填入，不需手動設定
📊
DB 索引設計說明
🔍
索引一：(examYear, examSession, examPaper) — 整卷查詢的主要路徑。前端作答頁一次撈整卷 80 題，這個索引讓查詢從全表掃描變成精確定位
🔍
索引二：(specialty, noteSlug) — 篩選特定科別、或找某篇筆記的相關考題時使用。兩個欄位合一個索引，節省 DB 資源

四個關鍵設計決策

這些不是隨便這樣做的——每個設計背後都有明確的理由。點開每張卡看完整說明。

互動 Demo：題目編號解碼器

輸入任意題號，看看它對應到資料庫的哪些欄位

格式：年-梯次-卷(3~6)-題號
🔢
題號是「算出來」的，不另存
115-1-3-1 由四個欄位組合產生

題目顯示給使用者看的編號 115-1-3-1，資料庫裡沒有這個欄位。它是由四個獨立欄位即時組合的：

examYear=115, examSession=1, examPaper="醫學三"(→代碼3), questionNumber=1
lib/quiz.ts → formatQuestionCode() → "115-1-3-1"
為什麼這樣做？ 如果另存一個 code 字串欄位，每次搜尋/排序都要解析字串，且必須保持同步。四個獨立的數字欄位更容易建索引、排序、範圍查詢（例如「撈出 115 年全部題目」只需 WHERE examYear=115）。
⚖️
更正題 "AC"、送分 "ALL" 的判分邏輯
考選部備註直接編碼進正解欄

考選部每梯次都會公布更正答案，例如「第1題答A或C者均給分」。我們把這個資訊直接存在 correctAnswer 欄：

// lib/quiz.ts
function isAnswerCorrect(correctAnswer, userAnswer) {
  if (!userAnswer) return false;
  if (correctAnswer === "ALL") return true;        // 送分題
  return correctAnswer.includes(userAnswer);       // "AC".includes("A") = true
}
為什麼這樣做？ 不需要另設 isAmended 欄位或 answers 陣列，一個字串就能表達所有情況。correctAnswer.includes(userAnswer) 這行程式碼對單答案（"B".includes("B")）和多答案（"AC".includes("A")）都能正確運作。
🔒
作答模式排除 correctAnswer
答案永遠只在 Server 端

兩個頁面對同一張資料表的查詢刻意不同：

// 作答模式（/test 頁）— 沒有 correctAnswer！
select: {
  id, questionNumber, questionText,
  optionA, optionB, optionC, optionD, optionE,
  questionType   // 就這樣，沒有正解
}

// 瀏覽模式（/exam 頁）— 包含正解
select: { ...上面全部..., correctAnswer }

判分在 POST /api/practice/submit：Server 端從 DB 取正解，再和使用者作答比對。

為什麼這樣做？ 有技術能力的用戶可能用 Chrome DevTools → Network tab 直接看 API 回應。只要作答時的回應根本不包含 correctAnswer，就算看到也沒用。這是「設計上不可能洩漏」，比「加密」更可靠。
🔗
UserAnswer 綁 PracticeSession（Cascade 刪除）
刪紀錄 → 錯題和雷達圖同步消失

每一筆作答（UserAnswer）都有一個 practiceSessionId，指向它所屬的測驗紀錄（PracticeSession）。schema 上設了 onDelete: Cascade：

// 刪除 PracticeSession 時，
// 屬於這份測驗的 UserAnswer 全部自動刪除

// quiz-stats.ts 的錯題/雷達邏輯：
// 只計入 practiceSessionId 非空的作答
where: { userId, practiceSessionId: { not: null } }

這樣設計讓「練習紀錄」真的有意義——使用者可以刪掉某次測驗，統計就自動重算。

為什麼這樣做？ 如果作答和測驗是獨立的（沒有 cascade），刪掉一次糟糕的練習後，那些答錯的記錄還會殘留在錯題本裡，造成資料不一致。Cascade 讓整個生命週期連動，永遠不會出現「孤兒作答紀錄」。

每年如何匯入新題目

全程四個步驟，完全冪等（重跑不會產生重複資料）。data/exams/ 是版本控管的權威來源，資料庫只是查詢的執行時副本。

1
解析 PDF → JSON（每卷執行一次，共 4 次）
從考選部下載試題 PDF（Q）與答案 PDF（A），每份卷分別執行解析腳本。
python3 scripts/parse-exam.py \ --q "116-1-醫學三-Q.pdf" --a "116-1-醫學三-A.pdf" \ --year 116 --session 1 --paper 醫學三 \ --out data/exams/116-1-3.json # 其他三份： # --paper 醫學四 --out data/exams/116-1-4.json # --paper 醫學五 --out data/exams/116-1-5.json # --paper 醫學六 --out data/exams/116-1-6.json
腳本會自動處理：換行黏接、全形轉半形、更正題（"AC"/"ALL"）。解析失敗直接報錯中止，不產出半套 JSON。
2
匯入資料庫（一次執行，可帶多個 JSON）
把四份 JSON 一次匯入資料庫。以 (examYear, examSession, examPaper, questionNumber) 做 upsert，重跑完全安全。注意：不會覆寫 specialty/topic/noteSlug（保留 AI 標記與人工修正）。
npx tsx --env-file=.env.local scripts/import-exam.ts \ data/exams/116-1-3.json \ data/exams/116-1-4.json \ data/exams/116-1-5.json \ data/exams/116-1-6.json # 輸出示例： # ✓ 116-1 醫學三：新增 80、更新 0 # ✓ 116-1 醫學四：新增 80、更新 0
3
AI 標記科別（產出標記 JSON → 套用到 DB）
先把題幹印出來供 AI 閱讀，再把 AI 的標記結果寫成 *.tags.json，最後用腳本套用到資料庫。
# 印出所有題幹供 AI 標記： python3 -c "import json; [print(q['questionNumber'],'|',q['questionText'][:90]) \ for p in [3,4,5,6] \ for q in json.load(open(f'data/exams/116-1-{p}.json'))['questions']]" # （把輸出貼給 AI，請它產出 116-1.tags.json） # 標記 JSON 完成後套用： npx tsx --env-file=.env.local scripts/tag-specialties.ts \ --tags data/exams/116-1.tags.json --year 116 --session 1
各卷科別範圍固定（考選部卷頭即標明）：
醫學三 = 內科各次專科 + 家醫 + 醫倫 ／ 醫學四 = 小兒 + 神經 + 精神 + 皮膚 + 醫倫
醫學五 = 外科各次專科 + 骨科 + 泌尿 + 醫倫 ／ 醫學六 = 婦產 + 復健 + 眼科 + 耳鼻喉 + 麻醉 + 醫倫
AI 不需要猜測卷別，只需從固定範圍內選最合適的科別，準確率很高。
4
驗證 + 收尾
確認題數正確、沒有未分類題目，然後 commit 進 git。
# 驗證：期望 題數=320（4卷×80）、未分類=0 npx tsx --env-file=.env.local -e " import('./lib/db').then(async ({prisma})=>{ const y=116,s=1; console.log('題數', await prisma.question.count({where:{examYear:y,examSession:s}})); console.log('未分類', await prisma.question.count({where:{examYear:y,examSession:s,specialty:'未分類'}})); process.exit(0) })" # 收尾：把 JSON commit 進 git（這是可重現匯入的權威來源） git add data/exams/116-1-*.json data/exams/116-1.tags.json git commit -m "feat: 匯入 116 年第 1 次考古題（醫學三~六，共 320 題）"
重要原則：data/exams/*.json commit 進 git 後，資料庫就算清空也能一行指令還原。JSON 是「真相」，DB 是「快取」。

六個可以做的進階功能

以下功能都能直接從現有 schema 實作，不需要改表結構。

🎯
依科別掌握度智慧出題
計算使用者各科答對率（isCorrect + specialty），自動把「掌握度低的科別」題目比例拉高。等於是個人化的弱點加強模式。
UserAnswer.isCorrect Question.specialty PracticeSession.filters
👀
Admin 低信心題審核佇列
Admin 後台直接撈 aiConfidence < 0.8 的題目（或 IS NULL 代表尚未標記），顯示題幹讓 Admin 快速確認或修改科別，一鍵更新。
Question.aiConfidence Question.specialty Question.topic
📖
筆記 × 考題雙向連動
當 noteSlug 有值時，讀筆記底部顯示「本章節相關考題」（最近 3 年），練習題可跳回對應筆記。這是 BearNotes 最核心的差異化功能。
Question.noteSlug Question.examYear Note.slug
📅
跨年度同章節出題趨勢
把同一個 noteSlug（章節）在不同 examYear 出現的次數做成折線圖，讓學生知道哪個章節近年越考越頻繁、哪個已經很少出現。
Question.noteSlug Question.examYear Question.examSession
🔁
間隔複習錯題本
以「每題最近一次作答時間」（answeredAt）為基準，實作 Spaced Repetition：答錯的題下次在 1 天後、3 天後、7 天後再出現，直到連續答對為止。
